Read this first — the spec targets a different frontend

The RBAC guide is written for a codebase that uses web/src/lib/types.ts and an api.me() call. Our project is a Next.js App Router app with different file names and an already-wired auth flow. This plan maps every recommendation onto our structure, and corrects two of the spec's assumptions that don't hold here:

correction The spec says our User type is missing roles and me() is never called. Not true here.

Our session User is models.User from the auth service (store/auth-store.ts:7), which already declares roles?: string[] and user_id?: string (types/api/auth.ts:686,688). And useMe() is called — but only to console.log the result (components/auth/auth-provider.tsx:37–42). So the gap isn't "add the field and the call" — it's nothing consumes roles or user_id yet, and the values are dropped before they reach any control.

scope boundary This governs the HFT resource family only.

Accounts, runs (live + paper), and HFT strategies come from the HFT API and carry the RBAC described here. Our /strategies list, however, is built from XALPHA / MFT data (hooks/api/use-strategies.ts, StrategyInfo has no owner_id) — a separate service with its own access model. Ownership work below applies to it only if backend adds an owner field. Treated as a non-goal here; see §6.

Everything below hides or reshapes UI to match what the API will allow. It never grants access; it prevents dead-end clicks and clarifies why something is missing. A None-scoped user must still be handled by real 403 handling, not by hiding alone.

02

The scope model

Every user has zero or more roles (strings from the external auth service). The backend reduces them to exactly one scope; when roles map to more than one tier, the highest wins. The ladder below is priority order, top = strongest.

All
admin
Sees and mutates every user's accounts, strategies, and runs. The only scope that ever touches another user's account or stops/deletes another user's run.
Lab
researcherresearch-leadstudentintern
Read-sharing only. Sees own resources plus the quant-lab roster's strategies and backtest/paper runs. Never accounts, never live runs. All writes stay owner-or-admin.
Own
pmtrader
Own resources only — the default for a mapped role.
None
defaultcontributortesterunmapped / empty
No access. Every gated endpoint returns 403. Must be handled explicitly — presence of no role is a valid state.
what Lab really grants

Accounts: never shared — Lab behaves exactly like Own. Strategies: read own + roster's; write is owner-or-admin. Runs: read own + roster's backtest/paper only — live is always excluded, even for lab-mates; write is owner-or-admin. In practice most role-aware behavior reduces to two checks: is the caller admin, and does owner_id === my user_id.

03

Delivery plan — four phases

Ordered by dependency: identity and ownership are foundations that phases 3–4 consume. Each phase ships independently and is testable on its own.

P1

Session scope & identity

effort · Sfoundation

Derive the caller's scope from roles and expose { userId, roles, scope, isAdmin } from one place, so no component re-implements the reduction.

  • Add lib/rbac.ts — mirror rbac.rs's three role arrays (ALL_ACCESS, LAB_ACCESS, OWN_ACCESS) and a resourceScope(roles) reducer that applies the priority order. Annotate it with the rbac.rs path — it's a deliberate second source of truth (Option A) until backend ships a derived scope (see §5).
  • Extend hooks/use-auth.ts to surface userId, roles, scope, and isAdmin off the store's user. The values are already in the store from the /auth/token exchange — this just reads them.
  • Verify /auth/token actually populates user.roles at runtime (the field is optional). If it comes back empty, repoint the existing useMe() to the authoritative HFT GET /api/auth/me (now in the generated User type) and feed its roles into the store — replacing the debug console.log.

P2

Ownership plumbing

effort · S–Mfoundation

Carry owner_id from the API through the row transforms so controls can compare it to the current user. It's stripped everywhere today.

  • Thread owner_id into the row shapes: LiveRunRow, PaperRunRow (add the field in lib/mock/*.ts, populate in lib/transform/runs.ts toLiveRunRow/toPaperRunRow) and EditorTab (toEditorTab in use-hft-strategies.ts). Accounts have no owner_id field on the payload — non-admins only ever receive their own, so ownership is implicit; only the admin cross-user view needs it (backend ask, §5).
  • Add helpers to lib/rbac.ts: isOwner(resource, userId) and canMutate(resource, { userId, isAdmin }) = isOwner || isAdmin.
  • Update mock fixtures so at least one run/strategy is owned by a different user id, so shared-vs-owned rendering is exercisable before touching the real API.

P3

Honest error semantics

effort · M403404422

Stop collapsing failures into one generic state. Each status means something specific — surface it.

  • Make the raw HFT helpers carry the server's message: apiGet/apiPost/apiPut/apiDelete in lib/api-client.ts currently throw ApiError(status, res.statusText) — read the JSON body like throwEnvelopeError does, so 422 business messages survive. Status is already preserved.
  • 403 No access to this family — session-level, same for every resource. Render a dedicated "you don't have access to this section" state and/or hide the nav entry once scope is known to be None. Don't rely on hiding alone.
  • 404 Not found — or hidden from you. On single-resource endpoints the API deliberately conflates "doesn't exist" with "someone else's." Render as not found, never "forbidden."
  • 422 Business rule rejected (e.g. deleting a paper/live run, stopping a non-running run). Show the server's message verbatim — it's caller-facing.
  • Add the missing isError branches: strategies/, live-trading/, and paper-trading/ pages read only isLoading today, so a failed query silently becomes the empty state. Model the branch on the existing accounts/venues list error states, and the status-aware precedent at venue-list.tsx:100.
  • Replace static mutation error strings (stop-run, launch, account CRUD, strategy create) with status-aware copy driven off ApiError.status.

P4

Role-aware surfaces

effort · M–Lconsumes P1–P3

Apply scope and ownership at the actual controls: nav, write buttons, and shared-resource badges.

  • Nav gating. Add adminOnly? / minScope? to NavItem (nav-config.ts) and extend the existing filter in sidebar.tsx:85 (which already filters by mode). None → hide gated sections; any admin-only surface → gate on isAdmin.
  • Hide — don't disable — write controls you can't use. Because Lab read- and write-visibility diverge, a lab-mate's strategy/run appears in lists but Stop/Edit/Delete would 404. Gate on canMutate: Stop-run (live-runs-table.tsx:161), strategy delete/edit (editors-bar.tsx, toolbar.tsx), account edit/delete (account-list.tsx — cross-user only under admin).
  • Badge shared resources. On the strategies and runs lists, compare owner_id to userId and render an "owned" vs "shared" chip so a lab-mate's item isn't mistaken for your own.
  • Live runs are never shared — keep them out of any "shared with lab" filter/tab; a lab-mate's live run won't appear at all, not even greyed out. Accounts are never shared — build no shared-account affordance.
  • Delete-run affordance is backtest-only. Paper/live return 422 for everyone, always. No delete button exists yet — when one is added, render it only when mode === "backtest".

05

Asks for backend

Recommended

Derived scope on /api/auth/me

Return scope: "all" | "lab" | "own" | "none" so the frontend never re-implements the reduction. Small change in routes/auth.rs / domain/user.rs; removes the second source of truth in lib/rbac.ts.

Conditional

Owner on HFT Account

Accounts expose no owner field. Only needed for the admin cross-user account view — without it, admins can't label whose account a row is. Non-admin flows don't need it.

Conditional

Owner on XALPHA StrategyInfo

Our /strategies list is MFT/XALPHA and has no owner_id. Ownership badges/gating there are impossible until it's added — otherwise that list stays outside this RBAC work (see §6).

06

Non-goals & risks

Explicitly out of scope

  • The XALPHA /strategies (MFT) list — different service, no owner field, separate access model.
  • Any "shared account" affordance — accounts are never shared under any scope.
  • Server-side enforcement — untouched; this is purely presentation.

Risks to watch

  • Mock mode masks RBAC. USE_MOCK defaults true and fixtures are single-owner — Lab/admin behavior can't be exercised until mocks vary owner_id (P2) or you test against the real dev API with a Lab-role and an admin token.
  • roles may be empty/unmapped → None. Never rely on scope-hiding alone; always handle 403 explicitly.
  • Rule drift. If lib/rbac.ts mirrors rbac.rs (Option A), the two can diverge — the derived-scope backend ask retires this risk.
verification Definition of done

A None user sees gated sections hidden and a clear 403 state, not empty lists. A Lab user sees lab-mates' strategies/backtest-paper runs badged as shared, with no write controls on them, and no live runs. An owner sees full controls. All checked against the real dev API — mock mode is not sufficient proof.
